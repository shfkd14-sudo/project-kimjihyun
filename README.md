> 이 문서는 다음 두 가이드의 **모든 내용을 정리한 개발 지침서**입니다.
> - `docs/guidebook.pdf` — "처음 시작하는 당신을 위한 플레이북: 코드가 아닌 '문제 해결'에 집중하는 법 & 기말과제 완벽 점검 가이드"
> - `docs/기말과제_웹앱개발.pdf` — "완벽한 웹앱을 짓는 설계의 기술: 기말 웹앱 프로젝트 기획안 제출 가이드 (프로그래밍기초)"

> 기능 제안, 코드 작성, 기획안 작성, 제출물 점검 시 이 문서를 기준으로 판단한다.

## 1. 핵심 철학 — "코드가 아닌 문제 해결"

- **우리가 만드는 건 멋진 코드가 아니라, 누군가의 불편을 줄이는 도구다.**
  우리가 매일 쓰는 앱들의 공통점: 멋있어서가 아니라 내 문제를 해결해주기 때문.
- **좋은 서비스는 '기술'이 아니라 '문제'에서 시작한다.**
  - ❌ (기술 먼저) "파이썬 배웠으니까 뭔가 만들어보자."
  - ✅ (문제 먼저) "내 룸메이트는 왜 매일 ○○ 때문에 짜증날까?"
- **합격 기획의 '한 문장' 공식:**
  > **[누가(사용자)] + [무슨 문제로(불편함)] + [어떻게 덜 불편해지나(해결책)] = 합격 기획**
  - 이 한 문장이 만들어지지 않으면, 아직 기획이 아니라 단순한 아이디어일 뿐이다.
- **스타트업이 망하는 이유 1위 (42%): "아무도 원하지 않는 걸 만들어서."**
  기술이 없어서가 아니라 **방향이 틀려서**다.
- **만들기 '전에' 진짜 수요를 확인하라.**
  - ❌ "이거 있으면 좋겠어?" → 다들 착해서 좋다고 함 (가짜 신호)
  - ✅ "너 지금 이거 당장 쓸래?" → 진짜 수요가 드러남
  - 드롭박스의 비밀: 코드 한 줄 없이 '3분짜리 작동 영상'만 올려 수요를 먼저 폭발시켰다.

## 2. 협업 모델 — "나는 설계자, AI는 시공업체"

- **사람(나) = 설계자, AI = 시공업체.**
  - 막연한 아이디어 → AI의 추측성 코딩 → 오류가 넘치는 코드와 멘붕
  - 선명한 8단계 기획안 → 타겟에 맞는 정확한 코드 → 작동하는 웹앱 완성
- **선명한 기획을 받은 AI는 '정확한 실행력'을 발휘한다.** (잘 그린 설계도를 받은 시공업체가 정확히 시공하듯)
- **기획안이 선명할수록 AI는 더 정확한 코드를 만든다.**
  기획안 작성은 서류 작업이 아니라, **디버깅 시간을 줄여주는 가장 확실한 무기**다.

## 3. '웹앱 서비스'의 합격 기준

- **기준 한 줄: 다른 사람이 브라우저에서 직접 써볼 수 있는가?** (Gradio, Streamlit 등)
- 판정 흐름:
  - 터미널에서 `print()`로만 결과가 나온다 → ❌ 작동하는 UI가 필수
  - 분석 보고서만 있고 사용자가 입력할 수 없다 → ❌ 사용자가 직접 쓰는 기능 필요
  - 다른 사람이 브라우저(Gradio/Streamlit 등)에서 직접 써볼 수 있다 → ✅ **합격! 완벽한 웹앱 서비스**

## 4. 개발 첫걸음 — 조심해야 할 4대 함정

1. **기능 욕심**: 로그인+AI+결제… 다 하려다 결국 하나도 못 끝냄.
2. **완벽주의 착각**: 아무도 안 보는 구석을 다듬다가 핵심 기능 미완성.
3. **한 방에 다 짜기**: 100줄 한 번에 쓰고 첫 실행 → 에러 폭탄.
4. **"데모 때만 되면 돼"**: 이상한 값 넣으면 와르르 무너지는 모래성.

## 5. 실력을 키우는 개발 마인드셋 7

1. **MVP부터 작게 시작하기**
2. **항상 '돌아가는 상태' 유지하기**
3. **메인 흐름 '한 줄기' 먼저 뚫기**
4. **사용자처럼 가혹하게 테스트하기** (빈칸 / 이상한 값)
5. **이름(변수/버튼) 직관적으로 짓기** (3주 뒤의 나를 위해)
6. **30분 룰**: 막히면 30분 뒤에 질문하기
7. **AI는 대리기사가 아닌 '조수'로** (이해하고 쓰기)

> **MVP 원칙:** "돌아가는 못난 앱 > 안 돌아가는 멋진 계획"
> 인스타그램의 진짜 시작: 잡탕 앱(Burbn)에서 사람들이 쓰는 '사진' 기능만 남기고 다 버려서 대박이 났다. **버리는 용기가 필요하다.**

## 6. 기획안 8원칙 (제출 본문의 뼈대)

> 새로운 양식이 아니다. **4주차 기획서 6단계의 '기말고사 업그레이드'**다.
> 기존 6단계(① 문제 정의 ② 사용자 ③ 핵심 기능 ④ 입출력 ⑤ 제약 ⑥ 테스트 시나리오)에
> **⑦ 프로토타입 증거(눈에 보이는 결과) + ⑧ AI 사용 내역(협업의 투명성)** 2개 블록만 더 얹는다.
> **새로운 양식을 외울 필요가 없다. 기존 6단계를 기말 수준으로 '확장'만 하면 된다.**

작성 시 따르는 8원칙 (원칙 1~3은 결국 **"누구의 어떤 문제를 해결하나?"**를 명확히 하는 단계다):

1. **프로젝트 제목** — 짧고 분명하게 핵심을 찌른다.
   - ✅ "기말 공부 시간 배분 도우미"  /  ❌ "멋진 과제 앱"
2. **문제 정의 (1~3문장)** — 무엇이 불편하고 왜 이 서비스가 필요한지 명확히.
   - 예: "시험 과목이 많아 공부 시간이 분산된다. 난이도를 반영해 우선순위를 보여주는 서비스가 필요하다."
3. **사용자 / 사용 장면** — 누가, 언제, 어떤 입력을 넣으며 쓰는지 생생하게.
   - 예: "대학생이 시험 2주 전, 과목명/시험일/난이도를 입력해 계획을 짤 때 사용한다."
4. **핵심 기능 3가지 '트리아지(응급 분류)'** — 욕심내지 말 것. (아래 6.1)
5. **입력 (Input)** — 사용자가 무엇을 넣는가. (예: 과목명, 시험일, 난이도, 목표점수)
6. **출력 (Output) + 제약/현실 조건** — 앱이 무엇을 돌려주는가 + 한계를 미리 선언. (아래 6.2)
7. **테스트 시나리오 2개** — 정상 케이스 + 경계/실패 케이스. **숫자와 입력값 필수.** (아래 6.3)
8. **프로토타입 증거 + AI 사용 내역** — 현재 상태를 보여주는 링크/스크린샷 **1개 필수**, 그리고 AI 협업 내역. (아래 6.4, 8)

### 6.1 핵심 기능 트리아지 (원칙 4)

| 우선순위 | 의미 | 예시 |
|---|---|---|
| **1순위 (Must-Have)** | 오늘 꼭 살아 있어야 할 기능. 이게 안 되면 앱의 의미가 없음. | 과목별 공부 시간 자동 배분 |
| **2순위 (Should-Have)** | 있으면 서비스가 훨씬 좋아지는 기능. | 날짜별 계획표 시각화 |
| **3순위 (Could-Have)** | 시간이 남으면 붙일 추가 기능. | 진도율 체크 및 알림 |

> 욕심내지 마세요! **1순위 기능 하나만 완벽하게 작동해도 훌륭한 웹앱**입니다.

### 6.2 입출력 설계와 '현실의 벽' (원칙 5~6)

- **입력(Input):** 사용자가 무엇을 넣는가? (예: 과목명, 시험일, 난이도, 목표점수)
- **처리(Process) = 제약/현실 조건:** 앱이 가진 한계를 미리 선언하라. **방어적인 설계가 더 안전하다.**
  - 예) 하루 최대 공부 시간은 8시간으로 제한
  - 예) 로그인 없이 사용
  - 예) 개인정보(주민번호 등) 입력 금지
- **출력(Output):** 앱이 무엇을 돌려주는가? (예: 과목별 총 공부시간, 하루 계획표)

### 6.3 작동을 구체적으로 증명 (원칙 7)

- **테스트 시나리오 2개 (숫자와 입력값 필수):**
  1. ✅ **정상 케이스**: (예) 과목 3개, 시험일 차이 5일 입력 → 수학이 가장 먼저 배정되는지 확인
  2. ⚠️ **경계/실패 케이스**: (예) 과목명을 빈칸으로 입력 → 오류 메시지가 정상적으로 뜨는지 확인

### 6.4 현재 프로토타입 상태 (원칙 8)

- **링크 또는 스크린샷 1개 첨부 필수.**
- **상태 보고는 정직하게:** "현재 입력 4개를 받아 출력하는 기능까지 구현. 달력 시각화는 아직 미구현."
  → **되는 것과 안 되는 것을 정직하게 구분하라.**

## 7. 제출 패키지 (4가지 구성품)

1. **기획안 본문** — 문제 정의부터 우선순위, 프로젝트의 뼈대를 담은 문서. **형식은 자유, 항목 이름은 유지. 본인 언어로 작성.**
2. **프로토타입 증거 (1개)** — 현재 어디까지 구현되었는지 보여주는 스크린샷이나 링크.
3. **테스트 시나리오 (2개)** — 정상 작동과 실패/경계 상황을 검증할 구체적인 계획.
4. **AI 사용 내역 요약** — AI에게 무엇을 시키고, 내가 무엇을 판단했는지 적은 투명한 기록. **프롬프트 2개 이상.**

> 추가로 **README** 필요: 문제 / 기능 / 실행방법 / 스크린샷 포함.

## 8. AI 협업 가이드 — '투명성'이 핵심

- 슬라이더의 양 극단은 모두 틀렸다:
  - ❌ "내가 다 코딩함 (AI 안 씀)" → 가능하지만 시간이 오래 걸림
  - ❌ "AI가 다 만들었어요" → **절대 금지. 본인의 언어와 판단이 없음.**
  - ✅ 정답은 가운데: **"나의 논리 + AI의 시공"**
- 기획안에 **최소 4줄 이상** 포함하라.
- **핵심 질문:** "어디를 AI에게 시켰고, 나온 결과를 보고 **내가 무엇을 판단(수정)했는가?**"가 반드시 보여야 한다.

## 9. 진단 매트릭스

### 9.1 내 주제는 얼마나 단단한가?

| 🎯 훌륭한 주제 | ☁️ 약한 주제 |
|---|---|
| 시험공부 계획기, 팀플 역할분배기, 동네 상권 분석기 | "멋진 AI 앱", "재밌는 게임", "무엇이든 추천해주는 서비스" |
| **Why?** 사용자와 상황이 선명하고 입출력이 명확함. | **Why?** 문제, 사용자, 검증 기준이 모두 흐릿함. |

> **11주차 과제를 이어가고 싶다면?** 가능하지만 아래 3개 중 **최소 2개는 반드시 바꿔야** 한다:
> 1. 사용자 / 2. 문제 정의 / 3. 기능 범위(의사결정 장면 추가)
> 예시: 단순 요약 앱 → 소상공인 창업 후보를 비교/결정 돕는 앱으로 업그레이드.

### 9.2 제출 전 가장 자주 빠지는 4가지 함정

| 자주 나오는 실수 (X) | 이렇게 수정하세요 (O) |
|---|---|
| **기능 과부하**: 로그인, 채팅, 지도, 대시보드 다 할게요! | 핵심 기능 **1개 + 보조 2개**로 다이어트하세요. |
| **검증 부재**: 테스트 = "잘 돌아감" | 구체적인 **입력값과 기대되는 결과**를 숫자/문장으로 명시하세요. |
| **UI 부재**: 함수 코드 조각만 제출함. | 최소한 하나의 **입력/결과창**을 가진 '웹앱 서비스 UI'여야 합니다. |
| **주도권 상실**: 기획서 전체가 영혼 없는 AI 문장 톤. | "왜 이 기능을 넣었는지" **본인의 목소리(1문장)**를 반드시 추가하세요. |

## 10. 기말 평가의 3기둥 (왜 기획안에 공을 들여야 하나)

이 기획안은 단순한 서류 과제가 아니라, 최종 기말 평가의 세 기둥을 직접 좌우한다.

1. **기획·문제정의** — 기획안의 1~4번 항목이 선명하면 이 점수는 이미 확보된다.
2. **작동·완성도** — 7~8번(테스트/프로토타입)을 미리 고민하면, 코드가 거칠어도 프로젝트 방향은 절대 흔들리지 않는다.
3. **개선 반복 + AI 투명성** — AI 사용 내역을 정직하게 기록하는 것 자체가 훌륭한 엔지니어링 역량으로 평가받는다.

## 11. 기말과제 셀프 점검 (제출 전, 단계별)

### 1. 기획 / 문제 정의
- [ ] 문제를 1문장으로 설명할 수 있는가?
- [ ] 사용자가 누구인지 명확한가?
- [ ] 핵심 기능 3개가 우선순위로 정리되었는가?
- [ ] 입력과 출력이 명확하게 연결되는가?

### 2. 기능과 흐름 (Input → Process → Output)
- [ ] 최소 1개의 '핵심 흐름'이 끝까지 동작하는가?
- [ ] 빈 입력, 잘못된 형식 등 '입력값 오류'에 대한 대응이 있는가?
- [ ] 결과가 사용자에게 이해되도록 설명 문구가 있는가?

### 3. 테스트와 개선 기록  (개선 기록 = 점수!)
- [ ] 테스트 시나리오 2개가 존재하는가?
- [ ] 실패 사례 1개와 대응 방법이 적혀 있는가?
- [ ] 최소 2회의 '개선 기록'이 작성되었는가?
- 개선 기록은 완성도의 증거다. ("A에서 문제가 생겨 B로 바꿨다")

### 4. 제출 패키지 확인
- [ ] 동작하는 앱 (링크 또는 실행 화면)

## 14. Claude 행동 지침 (이 프로젝트에서 개발할 때)

위 원칙을 BringBack 및 본 과제 작업에 항상 적용한다. 구체적으로:

- **MVP 우선**: 기능 추가를 제안하기 전에 트리아지(1/2/3순위)를 확인하고, 1순위 핵심 흐름부터 완성한다. 기능 과부하를 경계한다.
- **항상 돌아가는 상태 유지**: 한 번에 대량의 코드를 쓰지 말고 작은 단위로 작동을 확인하며 진행한다.
- **가혹한 테스트 포함**: 빈 입력·이상값에 대한 오류 처리를 기본으로 넣고, 정상 1개 + 실패/경계 1개 시나리오를 갖춘다.
- **직관적 네이밍**: 변수/함수/버튼 이름을 3주 뒤에도 이해되도록 짓는다.
- **웹앱 UI 지향**: 결과물은 `print()`나 코드 조각이 아니라, 브라우저에서 직접 써볼 수 있는 UI(예: Gradio/Streamlit)를 목표로 한다.
- **AI 투명성 유지**: 무엇을 어떻게 바꿨는지 명확히 설명·기록하여, 사용자가 'AI 사용 내역'과 '본인 판단/수정'을 작성할 수 있게 돕는다. 사용자의 언어와 판단을 대체하지 않는다.
- **개선 기록 남기기**: 변경 시 "왜 바꿨는지(A → B)"를 함께 남겨 개선 기록 작성을 돕는다.
